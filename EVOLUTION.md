# EVOLUTION.md

## 1. 你如何实现提示功能？

提示功能分三层实现。

**领域层（Sudoku）**：在 `Sudoku` 类中新增两个方法。`getCandidates(row, col)` 计算指定空格的候选数集合——从 {1..9} 出发，排除该格所在行、列、3×3 宫格中已出现的数字，剩余即为候选数。`getNextHint()` 遍历所有空格，找到候选数恰好为 1 的格子，返回其坐标和唯一候选值；若所有空格都有至少 2 个候选数，返回 null。

**会话层（Game）**：`getHint()` 直接委托给 `currentSudoku.getNextHint()`，`getCandidates(row, col)` 委托给 `currentSudoku.getCandidates(row, col)`。Game 作为统一入口，不添加额外逻辑。

**适配层（gameStore）**：`getHint()` 按优先级处理三种情况。若用户已选中某个空格，调用 `game.getCandidates()` 获取候选数列表，候选数唯一则自动填入，多个则展示列表供用户参考。若未选中空格但棋盘存在唯一候选数格子，自动填入并通过提示信息告知用户。若既未选中也无唯一候选数，提示用户先点击选中空格。提示信息通过独立的 `hintMessage` store 驱动，在 Controls 组件区域展示，用户落子或撤销时自动清除。

## 2. 你认为提示功能更属于 Sudoku 还是 Game？为什么？

核心计算能力属于 Sudoku，对外接口归属于 Game，两者通过委托协作。

候选数计算完全基于棋盘的纯数学规则（行、列、宫的已有数字约束），不涉及游戏状态、历史记录、用户操作序列等概念，因此计算逻辑天然属于 Sudoku。如果将来更换 Game 的实现方式（如改为多人对战），候选数计算逻辑无需任何修改。

但作为游戏功能入口，提示应通过 Game 暴露给 UI 层。理由有三：第一，单一入口原则——UI 层只与 Game 交互，不需要直接依赖 Sudoku；第二，扩展性——未来如果提示需要考虑历史状态（如"最近三次提示不要重复指向同一格"），Game 层可以直接访问自己的历史数据，无需修改 Sudoku；第三，关注点分离——Sudoku 管棋盘规则，Game 管游戏流程，职责边界清晰。

## 3. 你如何实现探索模式？

探索模式通过创建独立的 `ExploreSession` 对象实现，而非简单地在 Game 上添加布尔标志位。

**ExploreSession 的结构**：拥有独立的 Sudoku 副本（从进入探索时主棋盘深拷贝而来）、独立的历史栈（支持探索内 Undo/Redo）、失败路径记忆集合（记录曾导致冲突的棋盘状态）、冲突检测方法。

**完整生命周期**：进入探索时，`Game.startExplore()` 创建 ExploreSession 并深拷贝当前主棋盘作为起点快照。探索中所有操作都在 ExploreSession 的独立副本上进行，主 Game 完全不受影响。提交时，`commit()` 遍历所有空格，找出用户新填入的数字，逐一通过 `parentGame.guess()` 写入主 Game 的历史栈。放弃时，直接丢弃 ExploreSession 对象，主 Game 无需任何数据恢复。

**冲突检测与失败记忆**：每次探索操作后检查棋盘是否存在数字冲突。若冲突，当前棋盘状态的字符串表示被加入 `failedPaths` 集合，UI 显示冲突警告。当用户通过 Undo 回到之前失败的局面时，系统可提示"此路径已探索失败"。

## 4. 探索模式的本质是什么？

探索模式的本质是 **Game 创建一个临时子会话**，并结合了**局面快照与回滚机制**。

它不是简单地在 Game 上添加一个新状态标志位（那样会导致主棋盘和探索棋盘混杂，回滚困难），而是创建一个完全独立的 `ExploreSession` 对象。这个子会话拥有：

- **独立的棋盘副本**：从主棋盘深拷贝而来，所有探索操作都在副本上进行
- **独立的历史栈**：探索模式内的 Undo/Redo 不影响主 Game 的历史
- **明确的提交/放弃接口**：提供了清晰的操作语义

快照与回滚机制体现在：进入探索时自动创建起点快照（`startSnapshot`），提交时对比快照找出变更并合并，放弃时快照确保主 Game 可以零成本回滚到探索前的状态。

选择这种方案而非"Game 进入新状态"的原因：新状态方案需要在 Game 内部维护两套棋盘和一个状态机，会让 Game 的职责膨胀。而独立子会话方案保持了 Game 的核心简洁，将探索相关的复杂性封装在 ExploreSession 内部。

## 5. 主局面与探索局面的关系是什么？

### 5.1 是共享对象还是复制对象？

**是复制对象，不是共享对象。**

进入探索时，ExploreSession 的构造函数执行：`this.startSnapshot = parentGame.getSudoku().clone()`，以及 `this.exploreSudoku = this.startSnapshot.clone()`。这意味着探索棋盘和主棋盘是两个完全独立的对象，修改探索棋盘不会影响主棋盘。

### 5.2 是否会产生深拷贝问题？

每次进入探索都需要一次完整的深拷贝（81 个格子的二维数组）。对于 9×9 的数独棋盘来说，这个开销极小（几百字节的内存和微秒级的 CPU 时间），不会产生性能问题。深拷贝的好处是保证了主棋盘和探索棋盘之间零引用污染——任何一方的修改都不会意外影响另一方。

### 5.3 提交时如何合并？

`ExploreSession.commit()` 执行以下合并逻辑：

1. 获取探索棋盘的当前状态 `exploreSudoku.getGrid()`
2. 获取起点快照 `startSnapshot.getGrid()`
3. 遍历所有 81 个格子，找出满足条件的格子：起点快照中该格为空（值为 0），且探索后该格非空（已填入数字）
4. 对每个符合条件的格子，调用 `parentGame.guess({ row, col, value })`

合并按行列顺序依次写入主 Game。每次 `parentGame.guess()` 会自动追加到主 Game 的线性历史栈中，因此提交后用户可以在主 Game 中通过 Undo 逐步撤销探索提交的内容。

**注意**：如果用户在探索模式下修改了主棋盘已有的数字（覆盖了初始给定的数字），这些修改不会被合并——只合并起点为空、探索后非空的格子。这保留了初始题目的不可修改性。

### 5.4 放弃时如何回滚？

放弃探索时，`Game.endExplore(false)` 仅做一件事：将 `this.exploreSession` 设为 `null`。

因为探索模式下所有操作都在 ExploreSession 的独立副本上进行，主 Game 的三个核心属性——`currentSudoku`、`history`、`currentIndex`——在整个探索过程中从未被触碰。Store 层同步将 `isExploring` 设为 `false`，`exploreGrid` 设为 `null`，Board 组件自动切换回渲染主棋盘。**回滚成本为零**。

## 6. 你的 history 结构在本次作业中是否发生了变化？

### 6.1 是否仍然使用线性栈？

**主 Game 的 history 仍然是线性栈，没有变化。** 结构保持 Homework 1 的设计：`history` 数组按时间顺序存储每次操作后的 Sudoku 完整快照，`currentIndex` 指针标记当前位置。`undo()` 前移指针恢复到上一个快照，`redo()` 后移指针重新应用下一个快照，新操作会截断指针之后的所有 redo 快照。

### 6.2 探索过程是否拥有独立 history？

**是的，ExploreSession 拥有完全独立的历史栈。** 其结构与主 Game 的 history 一致（线性栈 + 指针），但完全隔离。进入探索时初始化为仅含起点快照的单元素数组，探索内的每次落子、撤销、重做仅在这段独立历史中操作。放弃探索时，这段历史被整体丢弃，不对主 history 产生任何影响。

### 6.3 提交后如何进入主 history？

提交时，`ExploreSession.commit()` 遍历所有用户新填入的格子，依次调用 `parentGame.guess()`。每次调用 `parentGame.guess()` 都会在主 Game 的线性历史栈末尾追加一个新的快照。因此，探索提交的内容会作为**一组连续的操作**出现在主历史栈的末尾，用户可以像撤销普通操作一样逐步撤销探索结果。

### 6.4 是否引入了树状分支？

**没有引入树状分支。** 树状分支意味着 history 需要维护多个分支指针，并在提交时决定合并策略（如 DAG merge）。这会显著增加实现和理解的复杂度，且作业要求中明确说明"不要求 DAG 合并"。

当前方案选择了更简洁的"复制 + 提交"模型：探索期间主 history 不变，探索结果作为一组连续的 `guess()` 操作批量写入主历史栈末尾。失败路径记忆（`failedPaths` 集合）作为辅助结构，不改变 history 的线性本质。

## 7. Homework 1 中的哪些设计，在 Homework 2 中暴露出了局限？

**第一，Game 缺少状态/模式管理能力。** Homework 1 的 Game 只有"正常游玩"这一种隐含状态。当需要引入探索模式（一种与正常游玩行为不同的模式）时，原有类结构无法直接支持，必须新增 `exploreSession` 属性和 `startExplore()`、`endExplore()` 等方法。如果在早期就将"模式"作为一等概念建模，探索模式的接入会更自然。

**第二，棋盘与历史耦合过紧。** Homework 1 中 `Game.currentSudoku` 既是当前棋盘也是历史的一部分，Undo/Redo 直接修改它。探索模式需要一个与主棋盘完全隔离的操作空间，这暴露了"棋盘"和"游戏会话"之间缺乏分离的问题。如果 Homework 1 就将棋盘状态和会话管理解耦，探索模式可以更直接地复用现有机制。

**第三，缺少局面快照与恢复的显式接口。** 虽然 `Sudoku.clone()` 可以创建快照，但 Game 没有提供"保存当前状态、稍后恢复"的语义化方法。探索模式恰好需要这种能力——进入时保存起点、放弃时恢复起点。如果 Game 早期就提供 `createSnapshot()` 和 `restoreSnapshot()` 方法，探索模式只需调用这些方法，而不需要在 ExploreSession 中重新实现快照逻辑。

**第四，Game 对外的接口粒度偏粗。** Homework 1 的 Game 只暴露了 `guess`、`undo`、`redo` 三个操作。Hint 和 Explore 功能分别需要"查询候选数"和"创建子会话"的能力，原始的接口集合无法覆盖这些需求。

## 8. 如果重做一次 Homework 1，你会如何修改原设计？

**引入模式枚举。** 在 Game 中使用 `mode` 属性区分 `'normal'` 和 `'exploring'` 两种模式。模式切换会影响 `guess()`、`undo()`、`redo()` 的行为——探索模式下操作路由到 ExploreSession，正常模式下操作路由到主棋盘。这样 ExploreSession 不需要通过 `parentGame.guess()` 这种反向依赖来提交结果，耦合度更低。

**将 ExploreSession 内建为 Game 的一部分。** 与其把探索作为"外部附加"的功能，不如在 Game 设计初期就考虑"可能需要进行临时尝试"的需求，将子会话管理作为 Game 的内建能力。这会让探索模式从"后期打补丁"变成"设计内的扩展点"。

**提取"可快照"接口。** 定义一个统一的快照接口，Sudoku 和 Game 都实现它。这样 Hint 的起点保存、Explore 的快照创建、甚至未来可能的"自动保存"功能，都可以用同一套机制完成，减少各处的重复深拷贝代码。

**先做 Hint 再回看设计。** 如果在 Homework 1 阶段就实现候选数计算，会自然地发现"有些局面无法通过唯一候选数推进"这一问题，从而更早地为 Explore 模式预留空间。Hint 和 Explore 本质上是"确定性推理"和"假设性尝试"两个互补的策略，在早期就一并考虑会让整体架构更协调。

**分离棋盘与会话。** 将 Sudoku（纯棋盘数据）和 GameSession（操作历史、模式管理）彻底分离。Sudoku 只负责数据读写和规则校验，GameSession 负责 Undo/Redo 历史、模式切换、探索子会话管理。这样每个类的职责更单一，测试更容易，扩展更方便。
